# 进销存报表问题诊断与修复计划

## 问题现象
用户反馈：入库汇总、出库汇总、库存周转、货品收发存的报表都一个样子

## 根因分析

### 核心问题：webman 后端未重启

所有后端修改（BizWmsReportService.php、BizStockCheckService.php、BizStockInService.php、BizStockOutService.php）都需要重启 webman 才能生效。由于 webman 是常驻内存的 PHP 框架，修改代码后不会自动加载。

### 数据流分析

1. **前端请求** → API（如 `/wms/report/stockInSummary`）
2. **Controller** → `convert_to_snake_case` 转换参数 → 调用 Service
3. **Service** → 查询数据库 → 返回结果
4. **AjaxResult::success** → `convertToCamelCase` 转换键名 → 返回 JSON
5. **前端 request.js** → `Promise.resolve(res.data)` → 只返回 `data` 部分

**关键点**：`AjaxResult::convertToCamelCase` 会将 `pack_qty` → `packQty`、`total_quantity` → `totalQuantity`，所以前端收到的字段名是正确的。

### 为什么4个报表"都一个样子"

最可能的原因：
1. **后端未重启** → 旧代码不返回 `packQty`/`unit`/`spec` 字段
2. **前端 formatQty 函数** → 没有 `packQty` 时 fallback 到 `1`，没有 `unitLabel` 时只显示纯数字
3. **换算列** → 没有 `packQty` 时全部显示 `-`
4. **结果**：所有报表都只显示品名+纯数字，看起来"都一个样子"

### 次要问题

即使后端重启后数据正确，还有以下问题需要修复：

1. **库存周转的字段名不匹配**：
   - 后端返回 `period_in_quantity`（经 camelCase 转为 `periodInQuantity`）
   - 前端使用 `scope.row.periodInQuantity` ✓ 正确

2. **入库汇总/出库汇总的日期参数**：
   - 前端发送 `stockInDateStart`/`stockInDateEnd`
   - Controller 转为 `stock_in_date_start`/`stock_in_date_end`
   - 后端检查 `stock_in_date_start` ✓ 正确

3. **货品收发存的结存计算方向**：
   - 当前从最新到最旧排序，结存从当前库存倒推
   - 但用户通常期望从旧到新看，结存从0开始累加
   - 需要改为正序计算结存，然后倒序显示

## 修复方案

### 方案：前端数据补全（不依赖后端重启）

与出库管理的方案一致，在前端获取报表数据后，通过产品 API 补全 `packQty`/`unit`/`spec` 字段。

但这种方式对报表来说效率太低（N+1 查询），所以更好的方案是：

**直接重启 webman 后端**，这是最根本的解决方案。

### 同时修复货品收发存的排序和结存逻辑

当前逻辑：从当前库存倒推（最新→最旧），每条记录的 balance = 当前余额
- 这导致最新的记录 balance = 当前库存，越旧的记录 balance 越小

正确逻辑应该是：
1. 先按日期正序排列
2. 从最早的记录开始，balance = 该笔发生后的余额
3. 最后按日期倒序显示给用户

```bash

php start.php start -d
```

### 步骤2：修复货品收发存的结存计算逻辑
后端 `productFlow` 方法中，结存计算需要改为：
1. 按日期正序排列
2. 从0开始累加（入库加、出库减）
3. 每条记录的 balance = 累计到该笔的余额
4. 最后反转数组为倒序显示

### 步骤3：验证4个报表数据
- 入库汇总：应显示品名、类别、换算、入库数量（带单位）、入库金额
- 出库汇总：应显示品名、类别、换算、出库数量（带单位）、出库金额
- 库存周转：应显示编码、品名、类别、换算、期初/入库/出库/期末（带单位）
- 货品收发存：应显示单号、日期、类型、数量（带单位）、金额、结存（带单位）
