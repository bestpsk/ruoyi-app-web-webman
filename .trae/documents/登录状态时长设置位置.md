# 登录状态时长设置位置分析报告

## 📋 任务概述
查找项目中所有与登录状态时长、Token过期时间、Session有效期相关的配置位置。

## 🔍 查找结果汇总

### **1. 后端核心配置 - Token过期时间（主要）**

#### 📍 文件位置：[Constants.php](file:///d:/fuchenpro/webman/app/common/Constants.php) 第14行
```php
const TOKEN_EXPIRE = 30;  // 单位：分钟
```
**说明：**
- 这是**最核心**的登录状态时长配置
- 当前设置为 **30分钟**
- 影响范围：
  - JWT Token的有效期（第33行：`'exp' => time() + $this->expire * 60`）
  - Redis中用户会话数据的过期时间（第80行：`$redis->setex($userKey, $this->expire * 60, ...)`）
- **修改方式**：直接修改此常量值即可调整全局登录状态时长

#### 📍 相关实现：[TokenService.php](file:///d:/fuchenpro/webman/app/service/TokenService.php)
```php
// 第19行：读取配置
$this->expire = Constants::TOKEN_EXPIRE;

// 第33行：JWT过期时间计算
'exp' => time() + $this->expire * 60,  // 30分钟后过期

// 第77行：Redis过期时间计算
$loginUser->expireTime = $loginUser->loginTime + $this->expire * 60 * 1000;

// 第80行：Redis设置过期时间
$redis->setex($userKey, $this->expire * 60, json_encode($loginUser->toArray()));
```

```javascript
import Cookies from 'js-cookie'
const TokenKey = 'Admin-Token'

export function setToken(token) {
  return Cookies.set(TokenKey, token)  // 使用默认Cookie设置

```
**说明：**
- 前端的Token存储本身**没有设置过期时间**
- 实际的过期控制完全依赖**后端返回的401未授权响应**
- 当后端判断Token过期时，前端会触发重新登录逻辑

## ⚙️ 如何修改登录状态时长

### 方案一：修改后端Token过期时间（推荐✅）

**步骤1**：打开 [Constants.php](file:///d:/fuchenpro/webman/app/common/Constants.php)

**步骤2**：修改第14行的 `TOKEN_EXPIRE` 值
```php
// 示例：修改为2小时
const TOKEN_EXPIRE = 120;  // 单位：分钟

// 示例：修改为8小时（工作日时长）
const TOKEN_EXPIRE = 480;

// 示例：修改为24小时
const TOKEN_EXPIRE = 1440;
```

**步骤3**：重启Webman服务使配置生效

### 方案二：同时调整Session超时（可选⚠️）

如果项目同时使用了Session认证，需要同步修改 [session.php](file:///d:/fuchenpro/webman/config/session.php)：
```php
'lifetime' => 8*24*60*60,  // 修改为8天或更长
```

## 🔐 安全建议

1. **生产环境推荐**：30分钟 ~ 2小时
   - 平衡用户体验与安全性
   - 可配合自动刷新Token机制（见TokenService第65-71行的verifyToken方法）

2. **开发/测试环境**：可适当延长至8小时或更长

3. **敏感操作**：即使Token有效，关键操作（如修改密码、支付）应要求重新验证身份

4. **Token刷新机制**：项目已实现自动续期功能
   - 在Token到期前20分钟自动刷新（第69行：`MILLIS_MINUTE_TWENTY`）
   - 用户无感知体验更好
