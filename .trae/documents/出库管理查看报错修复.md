# 出库管理查看报错修复计划

## 问题描述

**错误信息**: `TypeError: Cannot read properties of undefined (reading 'items')`  
**触发场景**: 点击出库管理列表的"查看"按钮  
**错误位置**: stockOut/index.vue:194

## 问题分析

### 根本原因

后端 `selectStockOutById` 方法返回的是**关联数组**，而 `AjaxResult::success()` 对关联数组的处理方式是将其字段**合并到响应顶层**，而不是放在 `data` 字段中。

### 数据流分析

**后端返回**:
```php
// BizStockOutService.php
return [
    'stockOutId' => ...,
    'stockOutNo' => ...,
    'items' => [...],
];
```

**AjaxResult::success() 处理**:
```php
// AjaxResult.php 第18-19行
if (self::isAssociative($data)) {
    $result = array_merge($result, $data);  // 合并到顶层
}
```

**实际响应结构**:
```javascript
{
  code: 200,
  msg: '操作成功',
  stockOutId: ...,      // ✅ 在顶层
  stockOutNo: ...,      // ✅ 在顶层
  items: [...]          // ✅ 在顶层
}
```

**前端期望**:
```javascript
{
  code: 200,
  msg: '操作成功',
  data: {               // ❌ 没有 data 字段
    stockOutId: ...,
    stockOutNo: ...,
    items: [...]
  }
}
```

**前端代码**:
```javascript
// stockOut/index.vue 第427行
form.value = response.data  // ❌ response.data 是 undefined
```

## 注意事项

**为什么入库管理没有这个问题？**

入库管理的后端 `selectStockInById` 返回的是 **Model 对象**，而不是数组：

```php
// BizStockInService.php
return $stockIn;  // 返回 Model 对象
```

`AjaxResult::success()` 对对象的处理是将它放在 `data` 字段中：

```php
} elseif (is_object($data)) {
    $result['data'] = self::convertToCamelCase($data->toArray());
}
```

所以入库管理可以使用 `response.data`。

**为什么出库管理返回数组？**

为了确保字段名正确转换为 camelCase（如 `out_target_type` → `outTargetType`），我们手动构建了返回数组。这导致 `AjaxResult::success()` 将其合并到顶层。
