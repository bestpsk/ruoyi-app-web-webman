# 角色管理权限系统分析计划

## 一、权限字符（role_key）的作用与机制

### 1.1 定义位置
- **数据库字段**：`sys_role.role_key` (varchar(100))
- **模型定义**：[SysRole.php:14](webman/app/model/SysRole.php#L14)
- **示例值**：`admin`, `common`, `test`, `admin,common`（支持多角色逗号分隔）

### 1.2 核心作用

#### 作用一：角色标识符
- 作为角色的唯一字符串标识，用于程序逻辑判断
- 区别于 role_id（数字ID），role_key 更具语义化

#### 作用二：权限校验依据
在 [PermissionService.php:45-52](webman/app/service/PermissionService.php#L45-L52) 中的使用：
```php
public static function hasRole(LoginUser $loginUser, $role)
{
    $userRoles = self::getUserRoles($loginUser);  // 获取用户的 role_key 列表
    return in_array($role, $userRoles);  // 判断是否包含指定角色
}
```

#### 作用三：角色权限集合构建
在 [SysPermissionService.php:23-30](webman/app/service/SysPermissionService.php#L23-L30) 中：
```php
foreach ($roleList as $role) {
    $roleKey = trim($role->role_key);
    if (strpos($roleKey, ',') !== false) {
        $roles = array_merge($roles, explode(',', $roleKey));  // 支持多角色
    } else {
        $roles[] = $roleKey;
    }
}
```

### 1.3 使用场景对比

| 场景 | 使用方式 | 示例 |
|------|---------|------|
| 后端接口权限控制 | `PermissionService::hasRole($loginUser, 'admin')` | 判断是否为管理员 |
| 前端按钮显示控制 | `v-hasRole="['admin']"` | 根据角色显示/隐藏按钮 |
| 多角色支持 | role_key = "admin,common" | 一个角色标识包含多个权限 |

### 1.4 特殊常量
- **超级管理员标识**：`Constants::SUPER_ADMIN = 'admin'`（见 [Constants.php:26](webman/app/common/Constants.php#L26)）
- 当用户拥有 admin 角色时，自动拥有所有权限

## 二、数据权限（data_scope）的区分机制

### 2.1 定义位置
- **数据库字段**：`sys_role.data_scope` (char(1))
- **默认值**：`'1'`
- **SQL定义**（来自 sql-backup.sql:756）：
  ```sql
  `data_scope` char(1) DEFAULT '1' COMMENT '数据范围（1：全部数据权限 2：自定数据权限 3：本部门数据权限 4：本部门及以下数据权限 5：仅本人数据权限）'
  ```

### 2.2 五种数据权限范围详解

#### 📊 权限级别对比表

| data_scope值 | 权限名称 | 可访问数据范围 | 适用场景 |
|-------------|---------|--------------|---------|
| **'1'** | 全部数据权限 | ✅ 所有数据（无限制） | 超级管理员、高层管理者 |
| **'2'** | 自定义数据权限 | 🔶 指定部门列表 | 跨部门管理者、项目负责人 |
| **'3'** | 本部门数据权限 | 🏢 仅当前部门 | 部门经理 |
| **'4'** | 本部门及以下数据权限 | 🏢➕ 子部门数据 | 区域总监、高级主管 |
| **'5'** | 仅本人数据权限 | 👤 仅自己创建的数据 | 普通员工 |

### 2.3 核心实现逻辑

#### 实现文件：[DataScopeService.php](webman/app/service/DataScopeService.php)

```php
public static function applyDataScope($query, $loginUser, $deptAlias = 'd', $userAlias = 'u')
{
    // 1. 超级管理员直接返回，不限制数据
    if ($loginUser->isAdmin()) {
        return $query;
    }

    // 2. 获取用户的所有角色
    $roles = $loginUser->user ? $loginUser->user->roles : [];
    
    // 3. 根据每个角色的 data_scope 构建查询条件
    foreach ($roles as $role) {
        $dataScope = $role['data_scope'] ?? '1';
        
        switch ($dataScope) {
            case '1': // 全部数据权限 → 不添加任何条件
                return $query;
                
            case '2': // 自定义数据权限 → 查询 sys_role_dept 关联的部门
                $deptIds = SysDept::join('sys_role_dept', ...)
                    ->where('sys_role_dept.role_id', $roleId)
                    ->pluck('sys_dept.dept_id')
                    ->toArray();
                $conditions[] = [$deptAlias . '.dept_id', 'in', $deptIds];
                break;
                
            case '3': // 本部门数据权限 → 限制为当前用户部门ID
                $conditions[] = [$deptAlias . '.dept_id', '=', $loginUser->deptId];
                break;
                
            case '4': // 本部门及以下 → 当前部门 + 子部门（通过 ancestors 字段）
                $conditions[] = function ($q) use ($deptAlias, $dept, $loginUser) {
                    $q->where($deptAlias . '.dept_id', $loginUser->deptId)
                      ->orWhere($deptAlias . '.dept_id', 'in', function ($subQ) use ($dept) {
                          $subQ->select('dept_id')->from('sys_dept')
                               ->whereRaw("find_in_set(?, ancestors)", [$dept->dept_id]);
                      });
                };
                break;
                
            case '5': // 仅本人数据 → 限制为当前用户ID
                $conditions[] = [$userAlias . '.user_id', '=', $loginUser->userId];
                break;
        }
    }
    
    // 4. 将所有条件用 OR 连接（多角色取并集）
    if (!empty($conditions)) {
        $query->where(function ($q) use ($conditions) {
            foreach ($conditions as $condition) {
                // ... orWhere 逻辑
            }
        });
    }
    
    return $query;
}
```

### 2.4 关键技术点

#### 1️⃣ 多角色权限合并策略
- 用户可能同时拥有多个角色（如：部门经理 + 项目负责人）
- **采用 OR 逻辑合并**：取所有角色数据权限的并集
- 示例：角色A可访问部门{1,2} + 角色B可访问部门{3} → 最终可访问{1,2,3}

#### 2️⃣ 自定义数据权限的实现
- 通过中间表 `sys_role_dept` 维护角色与部门的关联关系
- 在 [SysRole.php:24-27](webman/app/model/SysRole.php#L24-L27) 定义了关联：
  ```php
  public function depts()
  {
      return $this->belongsToMany(SysDept::class, SysRoleDept::class, ...);
  }
  ```

#### 3️⃣ 本部门及以下的层级查询
- 利用 `sys_dept.ancestors` 字段存储部门祖先链
- 使用 MySQL 的 `FIND_IN_SET()` 函数查询所有子部门
- 例如：部门ID=100，ancestors="0,100" → 可查到所有以 "0,100," 开头的子部门

### 2.5 实际应用案例

#### 使用位置：[SysUserService.php:43](webman/app/service/SysUserService.php#L43)
```php
public function getList($params)
{
    $query = SysUser::query();
    
    // 应用数据权限过滤
    if (!empty($params['data_scope']) && !empty($params['login_user'])) {
        $query = DataScopeService::applyDataScope(
            $query,
            $params['login_user'],
            'sys_dept',   // 部门表别名
            'sys_user'    // 用户表别名
        );
    }
    
    return $query->paginate();
}
```

## 三、权限系统的完整架构

### 3.1 两层权限控制体系

```
┌─────────────────────────────────────────────────────┐
│                   第一层：功能权限                     │
│  ┌─────────────────┐  ┌─────────────────┐           │
│  │  菜单权限(perms) │  │  按钮权限(perms) │           │
│  │  (sys_menu表)    │  │  (sys_menu表)    │           │
│  └────────┬────────┘  └────────┬────────┘           │
│           └──────────┬─────────┘                     │
│                      ▼                               │
│         PermissionService::hasPermi()                 │
│              校验用户是否拥有操作权限                    │
├─────────────────────────────────────────────────────┤
│                   第二层：数据权限                     │
│  ┌─────────────────────────────────────────────┐     │
│  │          data_scope (5种级别)                  │     │
│  │  全部 / 自定义 / 本部门 / 本部门及以下 / 本人    │     │
│  └────────────────────┬────────────────────────┘     │
│                       ▼                              │
│         DataScopeService::applyDataScope()            │
│              SQL查询时动态添加 WHERE 条件               │
└─────────────────────────────────────────────────────┘
```

### 3.2 权限字符 vs 数据权限 对比

| 维度 | 权限字符（role_key） | 数据权限（data_scope） |
|------|---------------------|---------------------|
| **作用对象** | 功能操作（能否做某事） | 数据范围（能看到哪些数据） |
| **控制粒度** | 接口/按钮/菜单级 | 数据记录级 |
| **校验时机** | 请求拦截时 | SQL查询时 |
| **实现方式** | 内存中的集合匹配 | 动态拼接SQL WHERE条件 |
| **典型值** | `admin`, `common` | `'1'`, `'2'`, `'3'`, `'4'`, `'5'` |
| **前端用途** | 控制 UI 元素显隐 | 通常不直接用于前端 |

### 3.3 典型业务流程示例

**场景**：部门经理查看用户列表

```
1. 用户登录 → Token中携带 userId, roles信息
       ↓
2. 请求 /system/user/list 接口
       ↓

       ↓

       ↓
7. 返回仅包含本部门的用户列表给前端
       ↓
8. 前端根据 response.data 渲染表格
```

## 四、总结与最佳实践建议

### 4.1 权限字符（role_key）设计原则
1. **语义化命名**：使用有意义的英文标识（如 `hr_manager`, `sales_staff`）
2. **避免频繁修改**：role_key 可能被硬编码在前端或配置中，修改需谨慎
3. **支持多角色**：可用逗号组合多个角色标识
4. **保持简洁**：建议使用小写字母和下划线

### 4.2 数据权限（data_scope）选择指南
- **超级管理员**：始终设为 `'1'`（全部数据权限）
- **跨部门岗位**：使用 `'2'`（自定义），配合 `sys_role_dept` 配置
- **部门负责人**：使用 `'3'` 或 `'4'`（视管理深度而定）
- **普通员工**：使用 `'5'`（仅本人数据）

### 4.3 注意事项
⚠️ **多角色场景**：当用户拥有多个角色且 data_scope 不同时，系统会取**权限并集**（OR逻辑）
⚠️ **性能考虑**：data_scope='4'（本部门及以下）涉及递归查询，在部门层级很深时可能影响性能
⚠️ **前后端一致性**：确保前端展示的数据与后端返回的一致（遵循"前端需要 response.data"规则）
